# 모임앱 서비스 방향 정리

## 1. 문서 목적

이 문서는 현재 구상 중인 모임앱의 문제 정의, 타겟 사용자, 핵심 기능, 운영 기능, MVP 우선순위, 이후 확장 방향을 정리한 기준 문서다.

앞으로 기획, 디자인, DB 설계, API 설계, 우선순위 판단 시 이 문서를 기본 레퍼런스로 사용한다.

---

## 2. 서비스 한 줄 정의

스포츠 및 동호회 모임을 위한 `탐색 + 운영 + 참여` 통합 플랫폼

단순히 모임을 찾는 서비스가 아니라, 운영자 입장에서는 회비/출결/회원관리까지 할 수 있고, 사용자 입장에서는 주변 모임 탐색, 참여, 기록 확인, 파트너 매칭까지 연결되는 구조를 목표로 한다.

---

## 3. 핵심 문제 정의

### 사용자 문제

#### 1) 모임 탐색

- 집 근처에서 활동하는 모임을 쉽게 찾기 어렵다.
- 지도 기반으로 주변 모임을 한눈에 보고 싶다.
- 종목, 지역, 연령대, 활동 인원 같은 기준으로 모임을 비교하고 싶다.

#### 2) 모임 신뢰도

- 분위기가 좋은 모임인지 미리 판단하기 어렵다.
- 모임 구성원의 연령대, 실제 활동 인원, 운영 스타일을 확인하고 싶다.
- 신규 가입 전에 “이 모임이 나와 맞는지” 가늠할 정보가 부족하다.

#### 3) 투명한 운영

- 회비가 어떻게 걷히고 어떻게 쓰이는지 알기 어렵다.
- 모임 운영이 폐쇄적이면 불신이 생긴다.

#### 4) 활동 확장

- 대회 참가 시 같이 나갈 파트너를 찾기 어렵다.
- 같은 종목, 비슷한 급수, 같은 지역의 사람을 연결해주는 기능이 필요하다.

### 운영자 문제

#### 1) 회원 관리

- 회원 회비 입금 여부를 일일이 확인하는 것이 번거롭다.
- 관리가 필요한 회원을 빠르게 찾기 어렵다.
- 회원 상태를 체계적으로 관리하기 어렵다.

#### 2) 운영 관리

- 출석 현황과 활동률을 파악하기 어렵다.
- 게스트 참석 이력을 따로 관리하기 번거롭다.
- 대진 구성, 휴식 조정, 게임 참여 균형을 수작업으로 맞추기 어렵다.

## 8. 이 서비스가 특히 잘 풀릴 수 있는 포지션

이 서비스는 단순한 “모임 만들기 앱”보다 아래 포지션에 더 강점이 있다.

### 1) 스포츠 동호회 특화

배드민턴, 테니스, 풋살처럼 반복 일정, 출결, 회비, 게스트, 대진, 파트너 매칭이 중요한 모임에 특화할수록 차별화가 생긴다.

### 2) 지도 + 운영 툴 결합

기존 서비스는 보통 모임 탐색까지만 강하거나, 내부 운영 도구까지만 강한 경우가 많다.
이 서비스는 `탐색`과 `운영`을 함께 묶는 게 핵심 차별점이다.

### 3) 신뢰도 정보 제공

연령대, 활동 인원, 출석률, 운영 투명성, 피드백 같은 요소를 노출할 수 있으면 신규 사용자의 가입 전환율이 높아질 가능성이 크다.

## 9. MVP에서 꼭 남겨야 할 것과 나중으로 미뤄야 할 것

### MVP에서 꼭 필요한 것

#### 사용자 측면

- 지도 기반 모임 검색
- 종목/지역 기반 필터
- 모임 상세 페이지
- 참여 신청
- 기본 프로필

#### 운영자 측면

- 회원 목록
- 출결 체크
- 회비 납부 여부 관리
- 게스트 관리

### MVP 이후로 미뤄도 되는 것

- 내부 메신저
- 익명 피드백 고도화
- 대진 자동 추천
- 코치 평가/후기
- 장비 거래
- 외부 광고/홍보

### 이유

MVP는 “정말 사람들이 이걸 운영 도구로 계속 쓸까?”를 검증하는 단계여야 한다.

따라서 초반에는 다음 질문에 답할 수 있는 기능이 우선이다.

- 사용자는 근처 모임을 실제로 찾고 참여하는가?
- 운영자는 출결/회비/게스트 관리를 이 서비스로 옮기고 싶어하는가?
- 모임 정보의 신뢰도 요소가 가입 결정에 영향을 주는가?

---

## 10. 추천 MVP 정의

### MVP 목표

`근처 스포츠 모임 탐색 + 모임 운영의 기본 관리`

### MVP 핵심 시나리오

#### 사용자

1. 집 근처 모임을 검색한다.
2. 지도와 리스트로 비교한다.
3. 모임 상세에서 분위기와 활동 정보를 확인한다.
4. 참여 신청을 한다.

#### 운영자

1. 모임을 개설한다.
2. 회원/게스트를 관리한다.
3. 출결을 체크한다.
4. 회비 납부 상태를 관리한다.

### MVP 핵심 화면

1. 홈
2. 검색
3. 모임 상세
4. 내 활동
5. 운영자 대시보드
6. 출결/회비/게스트 관리 화면

## 11. 서비스 발전 방향

### 방향 1. 탐색 서비스 강화

초기에는 사용자 유입을 위해 탐색 경험이 중요하다.

#### 발전 포인트

- 추천 모임
- 인기 모임
- 최근 활성화된 모임
- 연령대/활동량 기반 추천
- 사용자 위치 기반 정렬

### 방향 2. 운영 SaaS 강화

운영자가 계속 쓰게 만들면 이탈이 낮아진다.

#### 발전 포인트

- 월별 회비 자동 장부
- 출석률 통계
- 활동 회원/휴면 회원 분류
- 신규 회원 승인 플로우
- 운영 히스토리 로그

### 방향 3. 신뢰도 데이터 자산화

신뢰도는 가입 전환에 직접 연결되는 중요한 자산이다.

#### 발전 포인트

- 평균 활동 인원
- 연령대 분포
- 최근 출석률
- 운영 투명성 지표
- 신규 회원 만족도

### 방향 4. 스포츠 특화 네트워크 확장

모임 안에서 끝나는 게 아니라 종목 생태계로 확장할 수 있다.

#### 발전 포인트

- 파트너 구하기
- 대회 참가자 모집
- 지역 간 교류전
- 코치/레슨 연결
- 샵/장비 제휴

## 12. 제품 전략 제안

### 전략 A. “근처 운동 모임 찾기” 중심으로 시작

장점

- 사용자에게 바로 이해된다.
- 현재 구현 중인 지도 기반 탐색과 잘 맞는다.
- 빠르게 출시 가능하다.

한계

- 차별화가 약할 수 있다.
- 탐색 서비스만으로는 유지율이 낮을 수 있다.

### 전략 B. “운영이 편한 스포츠 모임 플랫폼”으로 시작

장점

- 운영자가 서비스에 깊게 묶인다.
- 재방문 빈도가 높다.
- 회비/출결/게스트 관리가 강한 차별점이 된다.

한계

- 초반 구축 범위가 넓어진다.
- 운영자용 UX까지 신경 써야 한다.

### 전략 C. “탐색 + 운영 기본형”으로 시작

추천 전략

- 사용자에게는 주변 모임 탐색을 제공한다.
- 운영자에게는 출결/회비/게스트 정도의 기본 운영 도구를 제공한다.
- 메신저, 대진 자동화, 광고 등은 나중으로 미룬다.

이 전략이 가장 현실적이다.
현재 앱 구조와도 잘 맞고, MVP 검증에도 적절하다.

- `Club` 또는 `Group`
  - 실제 운영 단위
  - 하나의 모임이 아니라 지속 운영되는 동호회 개념
- `Event`
  - 특정 날짜의 운동 일정
- `Attendance`
  - 출석 기록
- `Membership`
  - 회원 상태, 역할, 가입 승인 상태
- `GuestVisit`
  - 게스트 참석 이력
- `MembershipFee`
  - 회비 납부 상태
- `LedgerEntry`
  - 입금/출금 장부
- `PartnerPost`
  - 파트너 모집 글
- `Feedback`
  - 불편사항/운영 피드백

중요한 점은, 앞으로는 지금의 `Meeting` 하나가 모든 역할을 맡기보다:

- `Club`: 지속 조직
- `Event`: 개별 일정

으로 분리되는 방향이 더 자연스럽다는 점이다.

## 16. 현재 기준 추천 결론

이 서비스는 `스포츠 동호회 특화 탐색 + 운영 도구` 방향으로 가는 것이 가장 적합하다.

초기에는 아래 조합이 가장 현실적이다.

### 추천 1차 범위

- 근처 모임 탐색
- 모임 상세/참여
- 운영자용 회원/출결/회비/게스트 관리의 기본형

### 추천 2차 범위

- 신뢰도 지표
- 승인 기반 가입
- 파트너 찾기

### 추천 3차 범위

- 대진 관리
- 피드백/평판
- 레슨/코치
- 광고/제휴

이 순서가 제품 완성도, 차별화, 개발 난이도의 균형이 가장 좋다.
