# 2주 개발 로드맵

## 1. 목적

이 문서는 현재 코드베이스를 기준으로, MVP 검증에 가장 가까운 기능을 2주 안에 어떤 순서로 붙일지 정리한 실행 계획이다.

원칙은 아래와 같다.

- 당장 검증 가능한 기능을 우선한다.
- 구조 변경은 필요한 만큼만 한다.
- 운영자 핵심 편의가 빨리 나오도록 한다.

---

## 2. 2주 목표

2주 뒤 아래 상태를 목표로 한다.

- 사용자는 근처 모임을 찾고 참여할 수 있다.
- 사용자는 내가 참여 중인 모임과 일정 상태를 볼 수 있다.
- 운영자는 기본 대시보드에서 회원, 출결, 게스트, 회비 상태의 시작점을 볼 수 있다.

---

## 3. 주차별 목표

### 1주차 목표

- 내 활동 영역 만들기
- 모임 상세 신뢰 정보 보강
- 운영자 대시보드 초안 만들기
- `Club/Event` 전환을 위한 스키마 초안 정리

### 2주차 목표

- 출결 기본 기능
- 게스트 기록 기능
- 회비 상태 기본 기능
- 운영자 반복 사용 흐름 연결

---

## 4. 세부 작업

### Week 1

#### 1. 내 활동 페이지

목표

- 로그인 사용자가 내가 만든 모임과 참여 중인 모임을 한 번에 볼 수 있게 한다.

세부 작업

- `/me` 페이지 추가
- 내가 만든 모임 목록
- 참여 중인 모임 목록
- 예정된 일정 또는 가까운 날짜순 정렬

기대 효과

- 사용자가 다시 앱에 들어올 이유가 생긴다.

#### 2. 모임 상세 신뢰 정보 섹션

목표

- 모임 상세에서 가입 판단에 필요한 정보를 더 보여준다.

세부 작업

- 모임 종목 정리
- 활동 지역 정보 정리
- 모집 상태
- 평균 참여 인원 자리
- 연령대 자리
- 회비 안내 자리
- 가입 방식 자리

기대 효과

- 탐색에서 참여 전환으로 이어질 가능성이 높아진다.

#### 3. 운영자 대시보드 초안

목표

- 운영자가 필요한 정보를 한 화면에서 요약해서 본다.

세부 작업

- `/meetings/[id]/admin` 또는 이후 `clubs/[id]/admin`
- 총 회원 수
- 오늘 참여 인원
- 승인 대기 인원 자리
- 게스트 수 자리
- 회비 상태 요약 자리

기대 효과

- 운영 기능의 뼈대를 먼저 만든다.

#### 4. 데이터 모델 초안 정리

목표

- 현재 `Meeting` 구조에서 다음 단계 확장 방향을 팀 내 기준으로 확정한다.

세부 작업

- `Club`, `Membership`, `Event` 초안
- 관계 정리
- 마이그레이션 전략 초안

기대 효과

- 기능 개발 중 구조가 흔들리지 않는다.

### Week 2

#### 5. 출결 기본 기능

목표

- 운영자가 이벤트별로 출석을 체크할 수 있게 한다.

세부 작업

- 출석 데이터 모델 추가
- 이벤트 단위 출석 체크 UI
- 회원별 출석 상태 저장
- 최근 출석 요약 표시

기대 효과

- 운영자가 가장 자주 쓰는 기능을 확보한다.

#### 6. 게스트 관리

목표

- 게스트 참석 기록을 남기고, 반복 방문 여부를 볼 수 있게 한다.

세부 작업

- 게스트 엔티티 추가
- 이벤트별 게스트 등록
- 방문 횟수 표시
- 정회원 전환 후보 표시 자리

기대 효과

- 실제 스포츠 모임 운영 흐름과 가까워진다.

#### 7. 회비 상태 기본형

목표

- 회비를 장부 수준까지는 아니어도 납부 여부를 월 단위로 관리할 수 있게 한다.

세부 작업

- 월별 납부 상태 모델 추가
- 회원별 납부 여부 표시
- 미납 회원 필터

기대 효과

- 운영 툴로서의 차별점이 생긴다.

---

## 5. 우선순위 기준

작업 중 판단이 필요하면 아래 순서로 우선순위를 잡는다.

1. 운영자가 반복해서 쓰는가
2. 참여 전환에 영향을 주는가
3. 현재 구조로 빠르게 붙일 수 있는가
4. 이후 `Club/Event` 구조로 옮기기 쉬운가

---

## 6. 구현 우선순위 상세

### P0

- `/me`
- 모임 상세 신뢰 정보
- 운영자 대시보드 초안

### P1

- 출결 데이터 모델
- 출결 관리 화면
- 게스트 관리

### P2

- 회비 상태 관리
- 승인 대기 상태
- 가입 방식 고도화

---

## 7. 권장 실제 구현 순서

실제 작업은 아래 순서가 가장 안정적이다.

1. `/me` 페이지 구현
2. 모임 상세 확장
3. 운영자 대시보드 구현
4. 데이터 모델 1차 확장
5. 출결 구현
6. 게스트 구현
7. 회비 상태 구현

---

## 8. 성공 기준

2주 뒤 아래 질문에 “예”라고 답할 수 있어야 한다.

- 사용자가 다시 들어와서 내 모임 상태를 확인할 수 있는가
- 운영자가 최소한 출결과 게스트 관리를 앱에서 시작할 수 있는가
- 회비 관리의 시작점이 만들어졌는가
- 앞으로 `Club/Event` 구조로 확장 가능한 방향이 확보되었는가

---

## 9. 결론

다음 2주는 “기능을 많이 넣는 기간”이 아니라 “이 서비스가 탐색 앱을 넘어서 운영 도구가 될 수 있는지”를 검증하는 기간으로 보는 것이 맞다.

핵심은 다음 세 가지다.

- 사용자의 재방문 이유 만들기
- 운영자의 반복 사용 이유 만들기
- 다음 구조 확장을 위한 기준 만들기
